# customers
* preference_options 
    定義可選擇的偏好項目。
* customer_preferences 
    顧客與偏好項目的多對多關聯。
* customer_shipping_addresses 
    保存顧客的多組收件地址。
* customer_tags 
    顧客標籤主檔。
* customer_tag_assignments 
    顧客與標籤的多對多關聯。


## 關聯與資料流
customers
├─ 1:N customer_shipping_addresses
├─ N:M preference_options
│     └─ 透過 customer_preferences
└─ N:M customer_tags
      └─ 透過 customer_tag_assignments
### 關聯
* customers：會員資料的中心。
* customer_shipping_addresses：一位會員可以有多筆地址。
* customer_preferences：會員與偏好選項的多對多關聯表。
* customer_tag_assignments：會員與標籤的多對多關聯表。
### 資料流程
會員建立後：
1. 在 customers 建立基本身分資料。
2. 收件地址應寫入 customer_shipping_addresses。
3. 前台問卷從 preference_options 讀取可用選項。
4. 會員選擇結果寫入 customer_preferences。
5. 後台從 customer_tags 讀取可用標籤。
6. 管理員替會員加標籤時，寫入 customer_tag_assignments。
7. 查詢會員完整資料時，以上述關聯表向外組合資料。
---
刪除會員時：
1. customer_shipping_addresses：ON DELETE CASCADE，地址一起刪除。
2. customer_preferences：ON DELETE CASCADE，偏好關聯一起刪除。
3. customer_tag_assignments：ON DELETE CASCADE，標籤指派一起刪除。
4. 刪除偏好選項或標籤時則採 ON DELETE RESTRICT，只要仍被會員使用就不能刪除。



## 欄位的說明
### customers
* id                    識別碼，主鍵，預設由資料庫產生 32 字元 UUID
* name                  姓名
phone                   電話，NULL，資料庫沒有手機格式限制。
* email                 電子郵件，UNIQUE 限制。
birthday                生日，NULL
* registered_at         註冊時間，呼叫端必須自行提供。
tier                    等級代碼，NULL。
tier_name               等級顯示名稱，NULL。
* points                會員點數，預設 0，CHECK points >= 0。
* first_purchase_used   是否已使用首購優惠，預設 false。
* auth_provider         OAuth 登入來源，只允許 google、facebook、line。
* created_at            建立時間，預設 now()。
* updated_at            更新時間，預設 now()
    schema_copy.sql 沒有替本表設定自動更新 updated_at 的 Trigger。
avatar_url              頭像網址，NULL
* active                帳號是否啟用，預設 true。
deleted_at              軟刪除時間，NULL
    登入與一般查詢必須同時檢查 active = true及 deleted_at IS NULL。
*資料庫禁止直接 DELETE customers*
*idx_customers_auth_provider* 
*idx_customers_active_email*

### preference_options
* id            選項流水號，GENERATED BY DEFAULT AS IDENTITY，主鍵。
* type          偏好分類，只允許 style 或 equipment。
* code          選項代碼
* label         顯示給使用者的名稱
* sort_order    顯示順序，預設 0，CHECK sort_order >= 0。
* active        是否仍提供選擇，預設 true。
* created_at    建立時間，預設 now()。
* updated_at    更新時間，預設 now()。沒有自動更新 Trigger。
*(type, code) 必須唯一*
*(type, label) 必須唯一*
*idx_preference_options_type_active_sort*

### customer_preferences
* customer_id   識別碼，外鍵連到 customers.id。
* preference_id 偏好選項流水號，外鍵連到 preference_options.id。
*(customer_id, preference_id) 是複合主鍵*
*idx_customer_preferences_preference*

### customer_shipping_addresses
* id                地址流水號，GENERATED BY DEFAULT AS IDENTITY，主鍵。
* customer_id       地址所屬會員
* recipient_name    收件人姓名
* postal_code       郵遞區號，
* city 縣市
* district          鄉鎮市區
* address_line      街道、路段、門牌等詳細地址
* phone             收件人電話
* is_default        是否為預設地址，預設 false。
* created_at        建立時間，預設 now()。
* updated_at        更新時間，預設 now()，沒有自動更新 Trigger。
*每位會員最多只能有一筆 is_default = true 的地址*
*idx_customer_shipping_addresses_customer*
*idx_customer_shipping_addresses_one_default*

### customer_tags
* id                標籤流水號，GENERATED BY DEFAULT AS IDENTITY，主鍵。
* name              標籤名稱，UNIQUE。
* color             標籤顏色
* sort_order        顯示順序，CHECK sort_order >= 0。
* active            標籤是否啟用，預設 true。
* created_at        建立時間，預設 now()。
* updated_at        更新時間，預設 now()，沒有自動更新 Trigger。
*idx_customer_tags_active_sort*

### customer_tag_assignments
* customer_id   
* tag_id       
*(customer_id, tag_id) 是複合主鍵*
*idx_customer_tag_assignments_tag*



## 運作模式
### 軟刪除
* 會員刪除不是刪掉 customers 資料，而是：
    UPDATE customers
    SET active = false,
        deleted_at = now(),
        updated_at = now()
    WHERE id = ?;

## 可能的問題
* active 與 deleted_at 可能不一致
    建議加入狀態一致性約束 (不需要區分「停權」與「刪除」，可以統一規定)

* email UNIQUE 會阻止軟刪除後重新註冊
    如果允許重新註冊，應移除全表 UNIQUE，改成有效會員部分 UNIQUE INDEX。

* active_customers 沒有強制所有查詢使用

* 外鍵 ON DELETE CASCADE 與禁止硬刪除語意衝突
    建議未來統一政策。若永久禁止會員硬刪除，可考慮在新 migration 將這些外鍵改為 ON DELETE RESTRICT。 (當這筆資料仍被其他資料引用時，禁止刪除)

* tier 與 tier_name 可能重複且失去同步
    存在快取資料過期風險。

* registered_at 與 created_at 語意接近
    若兩者永遠相同，可以只保留其中一個。

* OAuth 身分資料不足

* 地址欄位只限制 NOT NULL，沒有防止空字串
    建議增加 btrim(column) <> '' 的 CHECK，並視需求驗證郵遞區號與電話格式。

* 地址只保證最多一筆預設地址 (可以直接設定只有備援)

* 偏好及標籤停用後，既有關聯仍會保留
    Mock 組合資料沒有過濾偏好或標籤的 active 狀態

* customer_preferences 與 customer_tag_assignments 關聯表缺少建立時間與操作者
    如果需要稽核，應補上時間與來源欄位

* updated_at 沒有統一自動更新機制
    更新資料時忘記同步設定 updated_at，時間就不正確。建議建立共用 Trigger Function，套用到：
        customers
        preference_options
        customer_shipping_addresses
        customer_tags

# articles
* article_content_blocks
    依排序保存文章內文區塊，可為文字、標題或商品嵌入區塊。
* article_related_products
    保存文章與推薦商品的多對多關聯及顯示順序。
* article_tags
    保存文章標籤。


## 關聯與資料流
articles
├─ 1:N article_content_blocks
│     └─ N:1 products（僅 block_type = product）
├─ N:M products
│     └─ 透過 article_related_products
└─ 1:N article_tags

### 關聯
* articles：文章主檔，保存文章的公開資訊、發布狀態與顯示設定。
* article_content_blocks：一篇文章有多個依 `sort_order` 排列的內容區塊；商品區塊可連結一項商品。
* article_related_products：文章與推薦商品的多對多關聯表；每篇文章中的推薦順序不可重複。
* article_tags：一篇文章可有多個標籤；同一篇文章不可重複使用同一標籤。
* article_dto_view：將文章主檔、標籤、推薦商品及內容區塊組合成前端相容的 JSON DTO。

### 資料流程
文章建立與發布：
1. 在 articles 建立文章基本資料；初始 `status` 預設為 `draft`。
2. 將段落與小標題寫入 article_content_blocks，`block_type` 為 `text` 或 `heading`。
3. 要嵌入商品時，建立 `block_type = product` 的內容區塊，並指定有效的 `product_id`。
4. 推薦商品另寫入 article_related_products，並提供每篇文章內唯一的 `sort_order`。
5. 標籤寫入 article_tags。
6. 發布文章時，將 `status` 設為 `published`，且必須同時提供 `published_at`。
7. 前端或 API 可透過 article_dto_view 取得整合後的文章 JSON；標籤按名稱、商品與內容區塊按 `sort_order` 輸出。
---
* 文章刪除：
DELETE articles
        ↓
article_content_blocks、article_related_products、article_tags
        ↓
由外鍵 ON DELETE CASCADE 一併刪除

* 商品刪除：
DELETE products
        ↓
若仍被 article_content_blocks 或 article_related_products 參照
        ↓
ON DELETE RESTRICT 阻止刪除


## 欄位說明
### articles
* id                    文章識別碼，由呼叫端提供。
* title                 文章標題
* category              文章分類
                        *idx_articles_category_published ON (category、published_at)*

* published_at          發布時間，NULL。
                        狀態為 published 時不得為 NULL。
* summary               文章摘要。
* cover_image_url       封面圖片網址，NULL。

* featured              是否為精選文章，預設 false。
                        *idx_articles_featured_published ON (featured、published_at)*

* status                文章狀態，預設 draft。
                        只允許 draft、published、archived。

* created_at            建立時間，預設 now()。
* updated_at            更新時間，預設 now()。
                        沒有自動更新的 Trigger。

### article_content_blocks
* id                    內容區塊流水號。
                        GENERATED BY DEFAULT AS IDENTITY。

* article_id            所屬文章識別碼。
                        ON UPDATE CASCADE、ON DELETE CASCADE

* sort_order            文章內顯示順序，必須大於等於 0。
                        同一篇文章不可重複。
                        *(article_id, sort_order) 複合 UNIQUE*

* block_type            區塊類型，只允許 text、heading、product。
* text_content          text 或 heading 區塊的內容。
                        block_type 為 text、heading 時必填，且 product_id 必須為 NULL。

* product_id            product 區塊所嵌入的商品識別碼。
                        block_type 為 product 時必填，且 text_content 必須為 NULL。
                        ON UPDATE CASCADE、ON DELETE RESTRICT。
                        *idx_article_content_blocks_product*

### article_related_products
* article_id            所屬文章識別碼。
                        ON UPDATE CASCADE、ON DELETE CASCADE。

* product_id            推薦商品識別碼。
                        ON UPDATE CASCADE、ON DELETE RESTRICT。
                        *idx_article_related_products_product*
                        
* sort_order            推薦商品顯示順序，必須大於等於 0。
                        同一篇文章不可重複。
                        *(article_id, sort_order) 複合 UNIQUE*

*(article_id, product_id) 是複合主鍵；同一篇文章不能重複關聯同一商品。*

### article_tags
* article_id            所屬文章識別碼。
                        外鍵至 articles.id；ON UPDATE CASCADE、ON DELETE CASCADE。
                        *pk_article_tags*
* tag                   標籤名稱，最多 100 字元，不可為空白字串。
                        *pk_article_tags*
                        *idx_article_tags_tag*

*(article_id, tag) 是複合主鍵；同一篇文章不能重複使用同一標籤。*


## 運作模式
* 文章基本資料與發布設定 > articles
* 文章段落、小標題及商品嵌入 > article_content_blocks
* 文章推薦商品 > article_related_products
* 文章標籤 > article_tags
* 前端相容 JSON > article_dto_view

### 文章狀態
* draft：草稿，可不填 `published_at`。
* published：已發布，必須填寫 `published_at`。
* archived：封存；資料庫未限制是否保留 `published_at`。

## 可能的問題
* 高風險：正式資料庫與前端存在兩套文章資料來源；前端目前讀取單一 `articles.json`，未讀取 article_dto_view。
* 高風險：目前前端與 Mock DTO 仍使用 author、authorAvatar、readTime，但 `latest_schema.sql` 的 article_dto_view 不提供這些欄位；前端改讀正式 View 前必須先移除或另行提供這些欄位。
* 高風險：前端 Mock 資料不受資料庫的狀態、發布時間與外鍵限制保護；正式 API 應只對外提供可見的 published 文章，並驗證商品仍有效。
* 中風險：articles.updated_at 沒有自動更新 Trigger；更新文章時必須由後端 Service 明確寫入。
* 中風險：`featured` 未限制只能套用於 published 文章；若有此業務規則，需由後端或新增約束保證。
* 中風險：article_dto_view 會輸出所有狀態的文章；若前台直接使用 View，查詢層必須加上 `status = 'published'` 條件。
* 低風險：標籤只禁止全空白，未統一大小寫、前後空白或同義詞；可能出現語意重複的標籤。
